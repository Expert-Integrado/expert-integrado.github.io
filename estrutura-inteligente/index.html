01 · O que é

Agente sem memória estruturada vira papagaio caro

Você passa o mesmo briefing toda semana, ele esquece toda sexta. A Estrutura Inteligente resolve isso do jeito mais simples que funciona: um bloco de memória versionado em Git.

Memória em arquivos

Tudo que o agente sabe vive em arquivos .md legíveis por humano — não numa janela de contexto que evapora.

Versionado em Git

Cada decisão, lição e mudança tem histórico. Formatou o PC? Clona e volta. A equipe inteira lê o mesmo bloco.

Onboarding automático

O aluno cola a URL do repo no Claude Code e o agente conduz tudo: entrevista, preenchimento e setup da VPS via SSH.

Independente de engine

Claude Code, OpenClaw, Cursor ou o que vier depois: qualquer ferramenta que leia arquivos usa o mesmo bloco.

Sem o bloco

  • Reexplica a empresa toda sessão
  • Decisões esquecidas entre conversas
  • Cada agente sabe uma coisa diferente
  • Setup morre se formatar o PC
  • Equipe sem acesso ao contexto

Com o bloco

  • O agente já sabe quem você é
  • Decisões versionadas em decisoes.md
  • Todos os agentes leem a mesma memória
  • Repo Git: clona e volta a operar
  • O time todo lê o mesmo arquivo
02 · O bloco

Três camadas de memória, uma pasta de identidade

A estrutura separa o que é pessoal do que é da empresa, o que é curado do que é captura bruta — e dá ao agente uma identidade própria em arquivos.

estrutura-inteligente/
├── memoria/            memória curada
│   ├── pessoal/        quem é você (gitignored)
│   ├── profissional/   empresa, áreas, skills
│   └── outros/         clientes, parceiros, ideias
├── memory/             daily notes brutas
│   ├── YYYY-MM-DD.md   1 sessão = 1 arquivo
│   └── feedback/       approve/reject em JSON
├── agente/             identidade do agente
│   ├── SOUL.md         personalidade
│   ├── IDENTITY.md     nome público, canais
│   ├── USER.md         quem é o dono
│   ├── TOOLS.md        MCPs, APIs, limites
│   ├── AGENTS.md       regras operacionais
│   ├── HEARTBEAT.md    tarefas recorrentes
│   ├── MEMORY.md       índice da memória
│   └── IMMUNE_SYSTEM.md
├── onboarding/         AUTO-ONBOARD, SETUP, skills
└── backups/            snapshot antes de mudança

memoria/Curada e navegável

Empresa, produtos, métricas, equipe, decisões e lições — mais uma pasta por área (vendas, marketing, atendimento…), cada uma com contexto, skills e rotinas. A parte pessoal fica fora do Git por padrão, via .gitignore.

memory/Bruta, por data

Captura raw de cada sessão, append-only. No fim do dia, o que importa migra pros arquivos curados. É o cânone que o OpenClaw indexa com memory_search.

agente/Quem o agente é

Os arquivos canônicos do padrão OpenClaw: alma, identidade, regras, ferramentas e agenda. Toda sessão nova começa lendo esses arquivos, na ordem definida em intro.md — e o que não está escrito neles, não existe.

CIEstrutura auditada

Uma GitHub Action valida a integridade do template a cada push: arquivos canônicos presentes, .gitignore protegendo a pasta pessoal, índices coerentes.

03 · Auto-onboard

O aluno cola uma URL. O agente faz o resto.

O repositório carrega o próprio manual de instalação — um playbook que o agente executa. Quem abre o repo num Claude Code diz uma frase ("configura esse repo aqui") e sai com um agente operacional 24/7 em 60–90 minutos.

0

Pré-flight check detecção automática

O agente abre memoria/pessoal/identidade.md: se ainda tem placeholders, o template é virgem e o onboarding dispara sozinho. Depois confirma os pré-requisitos (VPS contratada, SSH plugado, provider de modelo contratado) e testa a conexão SSH antes de qualquer coisa.

1

Entrevista 20–30 min

Uma pergunta de cada vez — nunca um formulário de 50 itens. Nome e personalidade do agente, quem é o dono, preferências de comunicação, empresa, produtos, a primeira área de atuação e a primeira skill. Cada arquivo preenchido é mostrado pro usuário validar antes de seguir.

2

Setup da VPS via SSH usuário só responde

O agente valida o Docker, clona o template na VPS, sobe o runtime OpenClaw em container, configura o provider de modelo, guia a criação de dois bots do Telegram no @BotFather, agenda 4 crons no runtime + 3 no host e instala as 3 camadas de resiliência do container. Todos os comandos rodam via SSH — o usuário não digita nada no servidor.

3

Validação e entrega prova real

O critério de pronto não é "instalou": é o heartbeat rodando de verdade e a mensagem chegando no Telegram do usuário. Só então o agente grava o marker de conclusão em MEMORY.md — e qualquer agente que abrir o repo depois sabe que o onboarding já foi feito.

04 · Engines

Braços mudam. Cérebro fica.

O bloco não pertence a nenhuma ferramenta. Ferramenta troca, modelo atualiza, agente reinicia — o contexto da empresa permanece em arquivo, independente de quem está rodando em cima.

Claude Code

Carrega por path explícito: uma linha no início da sessão manda o agente ler agente/intro.md, e o boot sequence faz o resto — SOUL, IDENTITY, USER, TOOLS, AGENTS, HEARTBEAT, MEMORY, na ordem.

OpenClaw

O template segue o cânone oficial: daily notes em memory/YYYY-MM-DD.md, indexáveis via memory_search, e os arquivos canônicos em agente/ — basta apontar o workspace do runtime pra essa pasta.

Cursor e outros

Qualquer ferramenta que leia .md no contexto da pasta funciona: aponta o context loader pra raiz do repo e referencia os arquivos por path quando precisar.

Boot sequence: como o agente acorda

O agente acorda zerado toda sessão — os arquivos são a continuidade dele. A cada sessão nova, ele lê os 8 arquivos de identidade na ordem definida, mais a daily note de ontem. Depois disso, sabe o suficiente pra operar sem pedir briefing.

1 SOUL · 2 IDENTITY · 3 USER · 4 TOOLS · 5 AGENTS · 6 HEARTBEAT · 7 MEMORY · 8 memory/[hoje].md

Princípio raiz

A memória não vive na janela de contexto, vive em arquivo.

"O que não tá escrito, não existe."

Por isso o fluxo termina sempre igual: o agente lê o que precisa, executa, e commita o resultado de volta.

05 · Sistema imunológico

70% de operar um agente 24/7 não é o agente — é o sistema imune

Sem isso, o agente quebra silenciosamente: cron some, credencial expira, container vira morto-vivo. O template já vem com as 5 práticas mínimas documentadas e agendadas no onboarding.

heartbeat

Cron a cada 30 min que checa o contexto e só fala quando há motivo real. Silêncio é OK — heartbeat que sempre fala vira ruído.

watchdog

Cron diário que audita os outros crons: cada um rodou na janela esperada? Retry 3x antes de alertar.

feedback

Approve/reject em JSON: o que o dono aprova ou rejeita vira regra consultada antes de cada sugestão nova.

audit

Auditoria periódica de integridade do próprio repo — a mesma checagem da CI, executável como skill.

backup

Snapshot antes de qualquer mudança estrutural, com template de rollback e smoke test pós-reversão.

Resiliência de container em 3 camadas

A versão mais recente do template fecha o caso do "morto-vivo": container marcado como Up por horas com o gateway interno morto, sem ninguém perceber. O onboarding agora instala três defesas independentes na VPS.

1 Watchdog systemd monitorando padrões de crash no log
2 Healthcheck Docker com probe HTTP a cada 30s
3 Autoheal: restart automático quando o Docker marca unhealthy
06 · Princípios

Duas regras que fazem o template render

O bloco é o esqueleto. O que transforma agente em operação são os hábitos que o template institucionaliza.

2x = metodologia

Toda tarefa que você pede mais de duas vezes vira skill: definir etapas, dar referências, executar uma vez supervisionado, transformar em checklist, salvar permanente na pasta da área. Esse é o multiplicador — sem isso, agente é só chat caro.

memoria/profissional/areas/[área]/skills/[skill]/SKILL.md

Esqueleto antes de cérebro

O template é a camada base de um caminho maior. Comece pelo bloco em .md; quando a memória crescer, plugue camadas mais sofisticadas por cima — sem jogar nada fora.
