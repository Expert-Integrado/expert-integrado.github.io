AI OS

A empresa descrita como uma especificação estruturada e governável — e operada com agentes de IA, de forma segura. Os dados estratégicos (pessoas, processos, decisões, regras) vivem num banco com controle de permissão; o código que dá vida a tudo vive no Git, com revisão por Pull Request. É o sistema operacional interno com que a Expert Integrado roda a própria operação.

01A ideia: descrever a empresa como uma spec

Em vez de espalhar a definição da empresa por planilhas, wikis e a cabeça das pessoas, o AI OS a guarda como dados estruturados num banco governado. A spec tem três eixos — e cada entidade é tipada por um contrato de schema, o vocabulário comum que elimina a ambiguidade e transforma documentação em especificação.

Pessoas

Quem é a organização: colaboradores, cargos, departamentos e papéis — tabelas como people, cargos, departments e roles.

Ferramentas

Os sistemas onde o trabalho acontece, catalogados na tabela saas — cada processo sabe em qual ferramenta cada passo roda.

Processos

Como o trabalho flui: quem executa cada passo, em qual ferramenta, e como o fluxo avança — sequencial ou por decisões. Tabela processes.

Contratos de schema

Cada eixo é tipado por um contrato guardado na tabela schemas. É o que torna a descrição precisa — e legível tanto por gente quanto por agente.

A spec vai além dos três eixos: o banco também guarda estratégia (linhas, produtos, riscos, decisões, gatilhos) e as entidades que são dado, não arquivo — skills, specs de LightApp e o cadastro dos Expert Brains de cada pessoa.

02Define aqui, executa lá

O AI OS governa a definição da empresa — não o dia a dia da operação. Ele não é uma engine de processos e, de propósito, não guarda o estado da execução. Cada processo declara onde é executado e controlado: a tarefa humana é projetada para a ferramenta de controle (ex.: ClickUp), e a parte da IA é entregue ao agente no momento certo, que age na ferramenta conectada via MCP.

Centralize a definição. Distribua a execução.

Não há plataforma central opaca rodando a operação: a informação estratégica vive central e governada, e a execução acontece distribuída — nas ferramentas externas e na máquina de cada colaborador, que orquestra e mentora a própria IA.

03Os building blocks

Abaixo da spec, blocos simples e combináveis dão vida à definição. O padrão que se repete em todos: o que é julgamento e definição é dado no banco; o que é motor e infraestrutura é código no Git.

Agentes

Executores de IA que conduzem etapas de processos dentro de regras. Catalogados como dado de governança e, quando funcionais, implementados como subagentes especializados — gestor de processos, desenvolvedor de LightApps, engenheiro do motor, DBA do banco.

Skills

Procedimentos detalhados que o agente executa de forma estruturada e confiável — criar um processo, montar um LightApp, fazer onboarding. A skill é dado, não arquivo: vive numa tabela do banco e é materializada localmente no início de cada sessão. A parte mecânica vira script determinístico, versionado no Git.

LightApps

Telas simples em que humano e agente operam juntos — o ponto de encontro visual. Cada app é um spec JSON renderizado por um runtime único sobre um catálogo fixo de componentes: o agente monta ou ajusta uma tela sem escrever código novo, e a mudança reflete ao vivo via Realtime.

Plugins (MCP)

A conexão com as ferramentas. Os dados governados chegam ao agente por um servidor MCP próprio, escrito em Rust, que opera o banco como o usuário logado — cada pessoa enxerga só o que a permissão dela permite. Ferramentas externas entram por MCPs remotos com OAuth.

Expert Brain

O grafo de conhecimento pessoal de cada colaborador, embutido como submódulo. Cada pessoa publica o próprio servidor na própria conta Cloudflare: o conhecimento tácito fica com o dono — separado, de propósito, do fato governado da empresa.

Scripts e hooks

A autoria mecânica (salvar skill, validar spec de LightApp) e a disciplina de sessão (sincronizar, proteger a main) são código determinístico no Git — o agente decide, o script executa igual toda vez.

04Como uma sessão de trabalho começa

Quando alguém abre o Claude no AI OS, uma sequência de verificações prepara o ambiente antes da primeira resposta — em ordem, porque as primeiras escrevem o estado que as seguintes leem. Tudo isso é código real dos hooks de início de sessão do repositório.

1

Sincroniza com a origem

Busca e aplica o que mudou no repositório remoto — toda sessão parte do estado mais recente do código.

2

Garante as dependências

Se faltam pacotes do runtime de LightApps na máquina, instala antes de qualquer uso.

3

Materializa as skills do banco

As skills de negócio vivem no Supabase; a sincronização as baixa para a pasta local de skills — um cache, fora do Git. Editar skill é operação no banco, não commit.

4

Confere o registro da pessoa

Se quem abriu a sessão ainda não está cadastrado como colaborador, o onboarding é conduzido na hora — nome, cargo, departamento, contatos.

5

Confere o Expert Brain e o trabalho pendente

Verifica o grafo pessoal de conhecimento, relata trabalho em aberto e Pull Requests aguardando revisão, e descobre os LightApps disponíveis.

6

Processo primeiro

Diante de um pedido, o agente consulta a tabela de processos antes de agir. Existe processo? Segue como guia. Não existe e a operação é recorrente? Oferece criar. E quando um processo falha ou está desatualizado, é atualizado na hora — cada execução deixa o processo melhor do que o encontrou.

06Governança: dois regimes, uma regra clara

A governança muda conforme a natureza do que muda. Dado e código têm mecanismos de controle diferentes — e é essa separação que permite ao agente registrar dados com liberdade sem nunca escapar da revisão onde ela importa.

DADOS DE NEGÓCIO

Supabase + RBAC

Pessoas, processos, estratégia, skills e specs de LightApp vivem no banco. A RLS libera leitura e escrita apenas para usuários autenticados com e-mail corporativo da empresa; de fora, nada.

A permissão é o controle — não há branch nem revisão por PR para dados. A integridade é garantida pelo próprio banco: chaves estrangeiras, campos obrigatórios, contratos de schema.

CÓDIGO

Git + Pull Request

Runtime de LightApps, servidores MCP, scripts, hooks e documentação chegam à branch principal via Pull Request. Hooks locais bloqueiam commit direto e force-push na main.

A revisão é o controle — e o início de cada sessão sincroniza com a origem, então todo mundo trabalha sobre o mesmo estado.

+Credenciais: nenhum segredo em arquivo

O princípio é único: MCP autentica pelo protocolo (OAuth) — cada pessoa autoriza uma vez no navegador, com o próprio login. Quando uma ferramenta só oferece chave de API, a chave vai para o cofre centralizado (Supabase Vault), criptografada, por usuário e protegida por RLS — nunca para um .env local.

A credencial é descriptografada só do lado servidor — o segredo nunca chega à máquina do usuário.
